## 一、产品定位

面向生活服务商家的轻量化对账工具：自动汇总团购订单、门店核销、收银流水三方数据，标记差异，替代人工对账。不做全功能收银/门店管理系统，只做"数据对得上、差异看得懂"这一件事。

> **背景说明**：公司业务规划（[DOC-002 业务规划](../doc/DOC-002-抖音体系内业务规划-小程序代开发能力验证-V2.md)第6节）里，对账工具只是"生活服务商家经营工具"矩阵的第一个模块，后续还规划了多门店经营看板、会员复购管理、AI经营建议、AI智能客服等模块，长期看是一个可扩展的经营数据平台方向。但**本项目的立项范围维持不变**，仍严格聚焦"对账"这一个可交付的 Demo，不在当前阶段扩大范围，避免功能蔓延拖慢第一个可演示成果的进度。远期模块规划以公司业务规划文档为准。

## 二、核心功能

1. **数据接入**：接入团购订单数据、核销记录数据、收银流水数据（三个独立数据源）。
2. **数据清洗与标准化**：统一三方数据的订单号、门店ID、金额、时间字段格式，处理缺失/异常值。
3. **三方比对引擎**：按订单号关联三方数据，识别以下差异类型：
   - 已核销但流水无记录（漏记流水）
   - 有流水但未核销（核销遗漏）
   - 核销/流水金额与订单金额不一致（多收/少收）
   - 同一订单被核销/记账多次（重复核销）
4. **差异清单与报表**：按门店、按时间维度展示差异汇总，支持按差异类型筛选查看明细。
5. **报表导出**：支持导出对账报表（用于向商家/内部演示）。

## 三、系统架构

采用两阶段数据接入路径（第一阶段验证算法，第二阶段验证自动化）：

- **第一阶段（静态验证，MVP）**：优先寻找1-2家友好测试商家，导出脱敏后的近一个月团购账单、门店核销日志、收银流水明细（Excel/CSV），作为真实数据源跑通清洗与对账逻辑；若短期内找不到愿意配合的测试商家，降级使用模拟数据兜底，不影响 M1-M3 节奏。**本阶段完全不接入任何实时 API（包括抖音开放平台），所有数据统一走文件导入**，把开发和验证重点放在清洗、匹配、异常检测算法本身，避免被接口权限/审核周期卡住进度。
- **第二阶段（自动化验证）**：Demo 验证通过后，评估接入抖音生活服务开放平台的订单/核销数据接口，配合 Cron 定时任务实现每日自动拉取对账（复用公司 B2B 数据采集器已有的 cron 调度经验）。

> ⚠️ 使用真实测试商家数据前，须与商家签署明确的数据使用授权/保密协议，且对外展示时必须如实说明"基于测试商家历史数据验证的能力 Demo"，不得包装为"已服务商家的正式案例"（红线要求，见 [项目立项书](DOC-001-项目立项书.md) 及业务规划文档第7.3节）。

```
数据源层
 ├─ 团购订单数据（第一阶段：测试商家历史账单导出 / 模拟数据兜底；第二阶段：抖音生活服务开放平台 API）
 ├─ 门店核销数据（第一阶段：测试商家核销日志 / 模拟数据兜底；第二阶段：开放平台 API）
 └─ 收银流水数据（Excel/CSV 导入，长期沿用此方式，不对接收银机硬件）
        │
        ▼
数据采集与清洗层（复用 B2B 数据采集器的多源数据管道设计思路）
 ├─ 数据标准化（字段映射、格式统一）
 └─ 数据校验（缺失值、异常值处理）
        │
        ▼
比对引擎层
 ├─ 按订单号关联三方数据
 └─ 差异规则匹配（4类差异场景）
        │
        ▼
存储层（SQLite，与现有数据系统保持一致的轻量化选型）
        │
        ▼
展示层
 ├─ 差异清单/报表可视化（Web页面）
 └─ 报表导出
```

## 四、数据模型（核心实体）

| 实体 | 关键字段 | 说明 |
|---|---|---|
| Order（订单） | order_id, store_id, product_name, amount, order_time | 团购订单原始数据 |
| VerificationRecord（核销记录） | order_id, store_id, verify_time, verify_amount | 门店核销数据 |
| CashFlowRecord（流水记录） | order_id(可选关联), store_id, amount, record_time | 收银流水数据，部分场景可能无订单号需人工/规则关联 |
| DiscrepancyRecord（差异记录） | order_id, store_id, discrepancy_type, expected_amount, actual_amount, detected_time | 比对引擎输出的差异结果 |

## 五、技术栈建议

沿用公司现有技术栈的成熟选型，降低引入新工具的学习和维护成本：

- 数据采集/清洗脚本：Python（与 B2B 数据采集器技术栈一致）
- 数据存储：SQLite（MVP阶段轻量、与现有系统一致；数据量增长后可平滑迁移至 PostgreSQL）
- 服务进程管理：PM2（与现有 SDR CRM 系统一致，便于统一运维）
- 前端展示：轻量 Web 页面（表格+图表），不引入重型前端框架，优先保证开发速度

> 公司技术栈统一为 Python + SQLite + PM2。

## 六、待确认事项（不作为已知条件，需后续核实）

- 抖音生活服务开放平台团购订单/核销数据的具体 API 接口、字段定义、鉴权方式——目前官方公开文档未披露详细的能力验证接口规范（见 [DOC-001](../doc/DOC-001-抖音开放平台服务商入驻-官方要求研究报告.md) 第三节信息缺口），M1-M3 先以真实测试商家历史数据/模拟数据代替，接口细节待人工登录服务商后台核实后再补充设计。**具体接口名称（如网传的 `poi.order` 等）未经官方一手文档核实，不作为设计依据，须实际登录服务商平台或联系招商团队确认后再写入正式方案。**
  - **2026-07-12 已核实（来自官方文档 SDK 概览 + 团购预约-订单确认接口页）**：
    - 接口 base 域名/路径规律：`https://open.douyin.com/goodlife/v1/...`（本地生活服务 OpenAPI 统一走这个前缀）。
    - 鉴权方式：请求头 `access-token`，通过"OAuth client_token 接口"用 `client_key`/`client_secret`（即 APPID/AppSecret）换取，具体换取接口的 URL/参数尚未核实到，见下方仍需核实项。
    - 官方只提供 Java / Go / NodeJS 三种语言的服务端 SDK，**没有 Python SDK**，Python 侧需按 OpenAPI 文档手写 HTTP 请求（`requests` 库直接调用即可，接口本身是标准 REST + JSON）。
    - 已核实到的具体接口示例：`团购预约-订单确认`（`POST /goodlife/v1/comprehensive/trade/order/confirm/`，权限 `life.capacity.comprehensive_reserve_confirm_order`）——但这是"商家确认/拒绝预约单"的操作接口，**不是**对账工具需要的订单/核销数据查询接口，且"团购预约"（到综预约品，先约后核销）和立项书里定的"团购订单核销"（团购券直接购买后到店核销）在业务模型上可能不是同一类目，需要确认清楚商家实际场景是"预约"还是"直接购买核销"，如果解决方案选错了要重新申请。
  - **仍待核实**：① `client_token` 换取 `access-token` 的具体接口 URL/参数；② 团购订单查询接口（列表/详情）；③ 核销记录查询接口。这三个是对账工具真正需要的接口，目前手里这两份文档都不是。
- 收银流水与订单号的关联规则：不同商家收银系统流水格式差异较大，是否需要人工映射规则或模糊匹配算法，待与真实商家数据联调后确定。
- 测试商家数据授权范围：具体需要商家提供哪些字段、脱敏到什么颗粒度，需与首批测试商家逐一协商确定。
- **二期数据源扩展的接口可得性（重要风险，需提前核实，不要等到二期才发现）**：
  - 美团点评的订单/核销接口大多不对外公开，多数需要服务商/ISV/合作伙伴身份才能调用，需提前确认是否需要单独走美团的服务商认证。
  - 主流收银系统（客如云、二维火、银豹、思迅、科脉等）各家 API 不统一，部分品牌可能没有开放接口，需要逐家调研，不能假设"都能接"。
  - 该风险的应对策略：MVP 阶段完全不依赖任何品牌的实时接口（见第三节第一阶段说明），二期扩展到具体某个数据源时，需先单独确认该数据源的接口可得性，再排入开发计划，避免接口调研阻塞整体进度。

## 七、非功能性需求

- **数据安全**：涉及商家经营数据和用户消费信息，需明确数据存储、传输的权限边界，不做超出对账范围的用户数据留存。
- **可扩展性**：数据模型设计需为后续"多门店经营看板""AI经营建议模块"预留扩展空间（复用同一套订单/核销/流水数据底座）。
